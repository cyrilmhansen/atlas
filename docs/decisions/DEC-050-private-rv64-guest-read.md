# DEC-050 - Add one checked guest read to the MIR RV64 probe

## Status

Accepted on 2026-07-13 (`rv64-guest-A`).

## Decision

Extend the temporary DEC-049 RV64 generator probe with the existing specialized
adjacent `is_sorted` control flow and one private `guest_load_i64(offset)`
import. The import treats MIR offsets as signed scalar values, rejects negative,
unaligned or out-of-bounds accesses, and reconstructs signed `i64` values from
explicit little-endian guest bytes.

The wrapper checks `element_count * 8 == byte_length` before generated code is
invoked. Empty, singleton, sorted-duplicate and first-inversion cases reuse the
same values as the native/interpreter/host-JIT correction test. An inconsistent
span must leave both the result sentinel and import-call count unchanged.

The RV64 probe keeps optimization level 2 and QEMU user mode. Its temporary
machine code must contain calls, conditional control flow and a return.

## Consequences

- This validates a generated RV64-to-host import call and the selected bounded
  offset semantics on the actual target ABI.
- The guest data remains one fixed-capacity, host-owned byte region. Offset zero
  is valid; no host pointer is exposed to MIR.
- The imported function and active region use process-local static state in a
  single-threaded test executable. They are not a public ABI or concurrency
  model.
- The probe validates read-only correction and bounds only. RV64 stores,
  mutation, multiple regions, allocation and semantic traces remain separate.
- Generated bytes remain temporary diagnostics, not archived executions or
  registry evidence.
- DEC-051 later adds a checked write import and `reverse` without changing the
  read-only contract established here.
