# DEC-049 - Probe MIR's RV64 generator as a standalone artifact

## Status

Accepted on 2026-07-13 (`riscv-gen-A`).

## Decision

Compile the pinned MIR core and generator for standard RV64GC LP64D, then run
that generator inside QEMU user mode. The probe builds one MIR `u64` addition,
generates native RV64 code at explicit optimization level 2, executes it, and
copies its exact bytes through the vendored synchronous observer.

The generated bytes are temporary. The cross-toolchain's `objdump` must decode
an addition and return, while the executed function must return 42 for `40 + 2`.
CI runs this separately from the existing compiler/ABI smoke test.

## Consequences

- This validates MIR's actual RISC-V generator, code allocator and execution
  under QEMU, rather than inferring support from a GCC-built RISC-V program.
- MIR selects its generator backend at compile time. This probe therefore
  cross-compiles and emulates the generator itself; Atlas does not expose a
  runtime target selector.
- The probe uses standard Linux RV64 LP64D and QEMU user mode. It establishes
  neither RV64ILP32, bare metal, Newlib nor a fantasy-computer machine profile.
- The generated code has no Atlas guest-memory imports, compact references,
  persistent representation or registry identity.
- Extending the probe to Atlas guest memory, imports or multiple regions remains
  a separate decision because it couples target ABI and runtime semantics.
- DEC-050 later accepts one private read-only guest import without changing the
  scalar probe or promoting its temporary artifact.
